# LLM-Judge Providers: Wiring the Second Eval Layer

The bundled judge runners (`llm_eval_runner.mts` for TypeScript repos, `llm_eval_runner.py` for Python repos) ship with three model backends compiled in. You never write provider code. You pick a backend with one environment variable and scaffold the matching `.env.example` block. That is the entire wiring job.

Scope of this document: backend selection and env scaffolding for the two runner templates. It does not cover eval methodology, judge prompts, scoring rubrics, adding a fourth backend, or CI secret management.

## Selection Mechanism

`EVAL_PROVIDER` picks the backend at runtime. Accepted values:

| `EVAL_PROVIDER` | Transport | Extra dependency |
|---|---|---|
| `anthropic-portkey` *(default)* | Portkey SDK → gateway `/v1/chat/completions` | `portkey-ai` (npm and pip) |
| `openai-compatible` | Raw HTTP POST, bearer auth | none |
| `anthropic-native` | Anthropic SDK → `/v1/messages` | Anthropic SDK for the language |

Notes:

- Unset means `anthropic-portkey`.
- The Python runner lowercases the value before matching; casing is forgiving there, but write it lowercase everywhere anyway.
- Change the backend by setting the variable in `.env` / `.env.example`, not by editing the default constant inside the runner source.

> Rationale: the runner templates are regenerated by the setup skill. Anything hand-edited inside them is lost on regeneration; env files survive.

Both runners lazy-import their SDKs. A repo that only ever uses `openai-compatible` installs zero extra packages — the Portkey and Anthropic imports never execute.

## Picking a Backend

| Your situation | Use |
|---|---|
| Anthropic models reached through a corporate Portkey gateway | `anthropic-portkey` |
| Direct Anthropic API access | `anthropic-native` |
| Anything else speaking the chat-completions protocol — hosted APIs, Azure-hosted deployments, vLLM, Ollama (`/v1`), LM Studio, Together | `openai-compatible` |

## Backend Details

### `anthropic-portkey`

Anthropic models behind a Portkey gateway. The gateway exposes the OpenAI chat-completions wire shape at `{base}/v1/chat/completions`, which is why the runner uses the Portkey SDK rather than the Anthropic one. Auth travels in an `x-portkey-api-key` header. The runner appends `/v1` to the gateway base URL if it is missing.

**The model string carries the routing.** Portkey resolves the upstream provider from the model id, not from headers. The id must be slug-qualified:

```
@<workspace-provider-slug>/<model>
```

A bare model name returns HTTP 400 complaining about a missing provider header. The fix is never to add headers — fix the model id. Discover valid slugs for your workspace:

```
GET {gateway}/v1/models
x-portkey-api-key: <key>
```

| Variable | Required | Meaning |
|---|---|---|
| `PORTKEY_API_KEY` | yes | gateway credential |
| `ANTHROPIC_BASE_URL` or `PORTKEY_GATEWAY_URL` | yes (either) | gateway base URL |
| `ANTHROPIC_MODEL` | yes | slug-qualified model id |

### `openai-compatible`

Plain HTTP POST with `Authorization: Bearer` against any chat-completions endpoint. No SDK involved. The runner appends `/chat/completions` to `OPENAI_BASE_URL`.

Local servers (Ollama, LM Studio, vLLM) usually ignore the auth header — set `OPENAI_API_KEY` to any non-empty placeholder anyway; the runner refuses an empty key.

| Variable | Required | Meaning |
|---|---|---|
| `OPENAI_API_KEY` | yes | bearer token (dummy ok for local servers) |
| `OPENAI_BASE_URL` | yes | endpoint base, without `/chat/completions` |
| `OPENAI_MODEL` | yes | model id as the server knows it |

### `anthropic-native`

Official Anthropic SDK against `/v1/messages` with `x-api-key` auth. The Messages API shape differs from chat-completions: `system` turns cannot ride in the message list, so the runner extracts them into the top-level `system` parameter. Use bare model ids here — no gateway slug. Set `ANTHROPIC_BASE_URL` only when routing through a proxy; omit it for direct API access.

| Variable | Required | Meaning |
|---|---|---|
| `ANTHROPIC_API_KEY` | yes | Anthropic API credential |
| `ANTHROPIC_MODEL` | yes | bare model id |
| `ANTHROPIC_BASE_URL` | no | proxy override only |

## Shared Knobs

| Variable | Default | Effect |
|---|---|---|
| `REPEATS` | `3` | runs each eval N times; outcomes collapse into a pass rate |
| `MAX_TOKENS` | `8192` | output cap for candidate and judge responses |
| *(temperature)* | `0`, hard-pinned | not configurable; applies to candidate and judge |

> Rationale: temperature 0 removes most sampling noise but not all of it. `REPEATS` absorbs the residual nondeterminism statistically instead of pretending it does not exist.

Truncation trap: when a candidate's answer hits `MAX_TOKENS`, the judge scores the missing tail as absent behavior — a false failure. If long-answer evals fail unexpectedly, raise the cap before doubting the judge.

## What the Setup Skill Must Generate

Two steps per repo (R2 — writes repo files):

1. Pick or confirm the default backend and record it as the `EVAL_PROVIDER` value.
2. Emit that backend's variable block into `.env.example` with placeholders only.

Credential hygiene contract:

- Never write a real token into any file. Not once, not "temporarily".
- Commit `.env.example` with `your-...` placeholders; `.env` stays git-ignored.
- The user copies `.env.example` to `.env` and fills values by hand.

Copy the matching block verbatim into the generated repo's `.env.example`:

**Portkey gateway**

```dotenv
EVAL_PROVIDER=anthropic-portkey
PORTKEY_API_KEY=your-portkey-api-key
ANTHROPIC_BASE_URL=https://your-gateway.example.com
ANTHROPIC_MODEL=@your-provider-slug/your-model-id
```

**OpenAI-compatible endpoint**

```dotenv
EVAL_PROVIDER=openai-compatible
OPENAI_API_KEY=your-api-key
OPENAI_BASE_URL=https://your-endpoint.example.com/v1
OPENAI_MODEL=your-model-id
```

**Direct Anthropic API**

```dotenv
EVAL_PROVIDER=anthropic-native
ANTHROPIC_API_KEY=your-anthropic-api-key
ANTHROPIC_MODEL=your-model-id
# ANTHROPIC_BASE_URL=https://your-proxy.example.com   # proxies only
```

Optionally append the shared knobs to any block:

```dotenv
# REPEATS=3
# MAX_TOKENS=8192
```

## Troubleshooting Quick Table

| Symptom | Backend | Fix |
|---|---|---|
| HTTP 400, "provider header required" | portkey | slug-qualify `ANTHROPIC_MODEL` (`@slug/model`); do not add headers |
| Unsure which slugs exist | portkey | `GET {gateway}/v1/models` with `x-portkey-api-key` |
| Long answers fail evals that look correct | any | raise `MAX_TOKENS`; truncation reads as missing behavior |
| Auth error against a local server | openai-compatible | set a non-empty dummy `OPENAI_API_KEY` |
| System prompt seemingly ignored | native | expected handling — runner lifts `system` turns to the top-level parameter; verify the eval file, not the runner |
| Import error for `portkey-ai` / Anthropic SDK | portkey / native | install the backend's package; only the selected backend's SDK is needed |

## Contract with the Runner Templates

The variable names and defaults above are the contract: the env-reading logic in `llm_eval_runner.mts` and `llm_eval_runner.py` must match this catalog exactly. If a template changes a variable name, a default, or an accepted `EVAL_PROVIDER` value, update this document in the same change.
