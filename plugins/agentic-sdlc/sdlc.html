End-to-end pipeline flow (/sdlc:start, /sdlc:autonomous)

Diagram 03 · v0.5

Lightweight task flow (/sdlc:task)

Six stages, all inline in the current conversation except a single subagent dispatch for code review. No worktree, no complexity scoring, no per-task evidence files. After Stage 6, the task enters maintenance mode and /sdlc:task sync reconciles spec.md + plan.md with subsequent inline changes.

From "I have a small task" to "branch ready" — without the ceremony

Use when you already know the task is XS / S / M. Each stage writes deterministic output under docs/superpowers/tasks/<date>-<slug>/. The transcript is the receipt — no evidence/<task>.json per task.

Why it's faster. The heavy pipeline dispatches one subagent per plan task (each loads project context + writes evidence/<task>.json). sdlc-task does TDD inline in the current conversation, so 0 subagent dispatches during implementation. Code review is the only model-heavy subagent call.
Diagram 04

Decision router · 3-step resolution

Every judgment gate hits this state machine. The order is intentional: cheapest path first. Routine evidence and green verification resolve without model calls.

How a gate becomes a verdict

Step 1 short-circuits in HITL mode. Steps 2–4 are autonomous-only. Escalation is a separate post-step that overrides the subagent verdict on low-confidence or matching risk flags.

Diagram 05

TDD evidence cycle (heavy pipeline)

Heavy pipeline enforces TDD via structured evidence files, not by inferring intent from commits or diffs. Each task subagent writes evidence/<task-id>.json; the pipeline validates it deterministically before final review. Note: /sdlc:task skips this cycle entirely — TDD runs inline in conversation and the transcript is the receipt.

Verification & Safety

AI-generated output goes through stricter verification, not lighter. Failures feed back into the implementation loop as tool results, with bounded retries before escalation.

sourceAgent outputDiff, commits, task evidence, and declared user-visible behavior.
formatFormat + lintPrettier, ESLint, Ruff, ktlint, or repo-native formatter and linter.
buildTypecheck + buildCompiler, package build, generated types, migrations, and static contracts.
unitUnit testsFocused red-green tests for the implementation unit.
integrationIntegration testsAPI, database, service boundary, and affected-package checks.
securitySAST + dependenciesSemgrep, npm audit, pip-audit, Trivy, Snyk, or equivalent project tooling.
qualityQuality scanSonarQube, CodeQL, coverage thresholds, duplication, and maintainability budgets.
reviewCode reviewFinal model review plus targeted check after fixes.
featureFeature verificationBrowser, CLI, API, screenshots, console, network, or smoke proof.
readyBranch readyAll required evidence is green and the branch is ready for handoff.
Failure path:Any failed check returns command output, screenshots, logs, or review findings to the implementation loop. Repeated or judgment-heavy failures route through the decision router.
